Executive Summary

SignalFlow brings reactive programming to Unreal Engine 5 with robust core architecture, automatic network replication, and full Blueprint integration.

üéØ The Problem

UI Polling Overhead: UMG property bindings poll every frame regardless of changes, wasting CPU cycles across all platforms.

Replication Complexity: Network replication requires extensive boilerplate code for each replicated property.

Blueprint Communication: Inter-Blueprint data sharing requires complex casting chains and event dispatchers.

GAS Attribute Limitation: Epic's Gameplay Ability System only supports float attributes (FGameplayAttributeData).

Actor Overhead: Each replicated value requires an Actor (~500+ bytes + tick), inefficient for data-heavy games on any platform.

üöÄ SignalFlow Solution

True Reactivity: UI updates only when data actually changes. Zero polling overhead.

Automatic Replication: Add a flag and values sync automatically via FFastArraySerializer.

Global Signal Registry: Any Blueprint can access any signal by name - no casting required.

Type-Agnostic Attributes: FSignalSet stores any type (int, bool, FVector, structs) - not just floats.

Actor-Decoupled Data: Singleton replicates all signals. 16-byte handles, event-driven. Zero per-value Actor overhead.

Quick Start

Get productive with SignalFlow in 5 minutes. No prior reactive programming experience required.

üß† Mental Model: 3 Core Ideas

üì¶

Signals are Values

Not events. A Signal<float> always has a current value. Read it anytime with .Value()

üîó

Dependencies Form a Graph

Computed signals track their inputs automatically. Change Health and HealthPercent updates.

‚ö°

Pull-Based with Caching

Values recompute only when needed. Same dependencies = same result. Zero redundant work.

üìã Your First Signal (5 minutes)

1

Include the Header

#include "SignalFlow.h"

This single header gives you access to all core SignalFlow functionality.

2

Create a Signal

auto Health = MakeSignal(100.0f); // Health.Value() == 100.0f

MakeSignal creates a mutable signal you can read and write.

3

Create a Computed Signal

auto HealthPercent = Computed([](float h) { return h / 100.0f; }, Health); // Automatically updates when Health changes

Computed signals derive from other signals. Dependencies are tracked automatically.

4

React to Changes

HealthPercent.Watch([](float pct) { UpdateHealthBar(pct); // Called ONLY when value changes });

Watch executes your callback whenever the signal value changes. Zero polling.

‚ö†Ô∏è When NOT to Use SignalFlow

SignalFlow excels at many things, but knowing its limits helps you make better architecture decisions.

‚ùå Frame-Critical Animation Logic

Don't call Signal::Value() inside AnimGraph evaluation. Use the AnimNotify bridge or cache values pre-frame.

‚ùå Competitive Shooter Abilities

Full GAS with prediction rollback is still better for competitive shooters. SignalFlow complements GAS, it doesn't replace it.

‚ùå Complex Custom Targeting Filters

AsSignal() works for standard radius+LOS. For complex multi-phase filters, use Execute() directly.

‚ùå Physics-Driven State

Don't wrap physics velocities in signals‚Äîthey change every frame. Use signals for gameplay state, not engine state.

üí° Rule of thumb: if data changes every single frame, it probably shouldn't be a signal.

‚ö†Ô∏è Common Mistakes

‚ùå Capturing Transient UObject*

Lambda captures UObject* that may be garbage collected before Watch fires.

// BAD: Widget may be destroyed Health.Watch([Widget](float v) { Widget->SetPercent(v); }); // GOOD: Use WeakCapture() helper auto Weak = WeakCapture(Widget); Health.Watch([Weak](float v) { if (auto* W = Weak.Get()) W->SetPercent(v); });
‚ùå Creating Computed() Inside Tick()

Creates new signal every frame, defeating the purpose of reactive updates.

// BAD: New signal every frame! void Tick() { auto X = Computed(...); } // GOOD: Create once, reuse void BeginPlay() { CachedComputed = Computed(...); }
‚ùå Using AsSignal() for Heavy Queries

FTargetingQuery::AsSignal() polls at fixed rate. Use Execute() directly for LOS-heavy scenarios.

// BAD: 10 LOS checks per second even when not needed Query.WithLOS().AsSignal(0.1f); // GOOD: Execute on demand in AI behavior tree TArray Targets = Query.WithLOS().Execute();
